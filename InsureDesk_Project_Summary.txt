  1. PROJECT OVERVIEW
================================================================================

  InsureDesk is a full-stack insurance agent portal designed for the US market.
  It enables individual agents, managers, and MGAs (Managing General Agents) to
  manage their entire book of business from a single interface.

  Core capabilities:
  - Lead & client pipeline management (Lead → Qualified → Proposal → Closed)
  - Policy lifecycle management (Draft → Submitted → Underwriting → Issued)
  - Multi-level commission tracking with immutable ledger
  - Agent hierarchy graph (MGA → Manager → Agent)
  - Task management with overdue alerts
  - Needs analysis worksheets
  - Analytics dashboard with production metrics
  - Renewal forecasting and monitoring
  - Role-based access control (Admin vs Agent)

  Target users:
  - Solo insurance agents
  - Agency managers / team leads
  - MGAs (Managing General Agents) overseeing multiple downline agents

  4. BACKEND SERVICES & APIs

  All endpoints are prefixed with /api. JWT Bearer token required on all
  routes except /api/auth/login and /api/auth/register.

  ── AUTH (/api/auth) ──────────────────────────────────────────────────────

  POST   /api/auth/register   Register a new agent account
                               Body: { name, email, password, role? }
                               Returns: { token, agent }

  POST   /api/auth/login       Authenticate and receive JWT
                               Body: { email, password }
                               Returns: { token, agent }

  GET    /api/auth/me          Return current agent profile from token

  ── AGENTS (/api/agents) ──────────────────────────────────────────────────

  GET    /api/agents           List all active agents (passwords stripped)
  GET    /api/agents/:id       Get one agent's profile
  PATCH  /api/agents/:id       Update name, phone, NPN, license_states

  ── CLIENTS (/api/clients) ────────────────────────────────────────────────

  GET    /api/clients          List clients (filterable: stage, agent_id,
                               lead_score_min, search, limit, offset)
  GET    /api/clients/:id      Get full client profile
  POST   /api/clients          Create new client/lead
  PUT    /api/clients/:id      Update client profile
  GET    /api/clients/:id/policies    All policies for a client
  GET    /api/clients/:id/activities  Activity timeline for a client

  ── PRODUCTS (/api/products) ──────────────────────────────────────────────

  GET    /api/products         List all insurance products (filterable)
  GET    /api/products/:id     Get product detail with eligibility rules
  POST   /api/products         Create new product (admin)
  PUT    /api/products/:id     Update product

  ── POLICIES (/api/policies) ──────────────────────────────────────────────

  GET    /api/policies         List policies (filterable: status, client_id,
                               agent_id, renewal_window in days)
  GET    /api/policies/:id     Get policy + status history
  POST   /api/policies         Create policy (runs eligibility validation:
                               age, income, premium min/max checks)
  PUT    /api/policies/:id     Update premium or documents
  POST   /api/policies/:id/transition  Advance FSM state:
                               Draft→Submitted→Underwriting→Approved→Issued
                               On "Issued": creates commission rows in both
                               commissions table AND commission_ledger via
                               BFS upward traversal of agent_hierarchy

  Policy FSM (Finite State Machine):
  Draft ──► Submitted ──► Underwriting ──► Approved ──► Issued ──► Lapsed
                    └──────────────────────────────► Rejected

  ── COMMISSIONS (/api/commissions) ────────────────────────────────────────

  GET    /api/commissions        List commissions (filterable: agent_id,
                                 event_type, date range)
  GET    /api/commissions/summary  Aggregate totals: total, sale, renewal,
                                   count — for current agent or all (admin)
  GET    /api/commissions/forecast Projected future commissions from:
                                   • Pending applications (probability-weighted
                                     by pipeline stage)
                                   • Upcoming renewals within 90 days

  ── ACTIVITIES (/api/activities) ──────────────────────────────────────────

  GET    /api/activities         List all activities (filterable: client_id,
                                 activity_type, agent_id)
  POST   /api/activities         Log a new activity (call, note, meeting,
                                 follow_up, status_change, etc.)

  ── TASKS (/api/tasks) ────────────────────────────────────────────────────

  GET    /api/tasks              List tasks (filterable: agent_id, status,
                                 priority, due_before, due_after)
  POST   /api/tasks              Create task with client/policy linkage
  PATCH  /api/tasks/:id          Update task (status, priority, due_date, etc.)
  DELETE /api/tasks/:id          Delete a task

  ── ANALYTICS (/api/analytics) ────────────────────────────────────────────

  GET    /api/analytics/summary  Full production metrics:
                                 • Policies issued MTD/YTD + prior period
                                 • Premium written + commissions earned
                                 • Pipeline conversion rates by stage
                                 • Avg days in each stage
                                 • Activity metrics (calls, meetings, notes)
                                 • Top clients by premium
                                 • Renewal forecast (30/60/90 days)
                                 • Commission trend (last 12 months)

  ── NEEDS ANALYSIS (/api/needs-analysis) ──────────────────────────────────

  GET    /api/needs-analysis/client/:id  All worksheets for a client
  POST   /api/needs-analysis             Create new needs analysis:
                                         • Coverage gaps, goals, risk score
                                         • Auto-computes recommended products
                                         • Logs as activity

  ── HIERARCHY + LEDGER (/api/hierarchy) ───────────────────────────────────

  GET    /api/hierarchy/graph             All agents as nodes + all active
                                          hierarchy edges. Earnings included
                                          only for visible agents (visibility-
                                          enforced at SQL level, not UI).

  GET    /api/hierarchy/agent/:id         One agent's direct uplines + downlines

  POST   /api/hierarchy/link              Create a hierarchy relationship
                                          (admin only). Cycle detection via
                                          recursive CTE before insert.
         Body: { upline_agent_id, downline_agent_id, override_percentage }

  DELETE /api/hierarchy/link/:id          Soft-delete (is_active=0) a link
                                          (admin only)

                                          agent sees own rows only; upline
                                          sees own + all downline rows; admin

================================================================================
  7. AUTHENTICATION & SECURITY
================================================================================

  JWT (JSON Web Token):
  - Algorithm: HS256 (HMAC-SHA256)
  - Payload: { agent_id, email, role, name }
  - Expiry: 7 days
  - Stored in: localStorage (key: insuredesk_token)
  - Sent as: Authorization: Bearer <token>

  Password Storage:
  - hashlib.sha256(password.encode()).hexdigest()
  - No salt (suitable for demo; production should use bcrypt/argon2)

  Role-Based Access Control:
  - "admin" role: sees all agents' clients, policies, commissions
  - "agent" role: sees only own data
  - Enforced in SQL queries (WHERE agent_id=? for agents)
  - Hierarchy visibility enforced via recursive CTE (not UI masking)

  Frontend Route Protection:
  - AuthGuard component wraps all routes
  - Redirects to /login if no token present
  - Token expiry results in 401 → toast error (manual re-login required)

================================================================================
  8. COMMISSION ENGINE
================================================================================

  When a policy transitions to "Issued" status, two things happen atomically:

  Step 1 — Legacy commissions row (backward-compatible):
    INSERT INTO commissions: one row for the writing agent at base rate

  Step 2 — Ledger BFS engine (compute_and_insert_ledger):

  Algorithm:
  1. BASE entry: writing agent earns (premium × base_rate%) — visibility SELF
  2. BFS upward through agent_hierarchy:
     - Start from writing_agent_id
     - visited set prevents cycles
     - For each direct upline found:
       a. Check commission_rules for product+level-specific override %
       b. Fall back to agent_hierarchy.override_percentage if no rule
       c. Compute amount = premium × pct% (using Python Decimal, 4 dp)
       d. INSERT OVERRIDE row — visibility DOWNLINE
     - Add upline to frontier, increment depth
     - Stop at MAX_LEVELS = 5 or when no more uplines found
  3. All inserts share the same uncommitted transaction as the
     commissions row — fully atomic (all succeed or all roll back)

  Example with 4-agent hierarchy (pol-XXX, premium=$1000):

    Sofia Park (agent-005, writing agent):
      └─► LEDGER: BASE  level=0 10.00% $100.0000 (visible to Sofia)

    Rachel Chen (agent-003, L1 upline, 3% override):
      └─► LEDGER: OVERRIDE level=1 3.00% $30.0000 (visible to Rachel + uplines)

    Alex Morgan (agent-001, L2 upline, 5% override via agent_hierarchy OR
                 2.5% via commission_rules for Whole Life Advantage):
      └─► LEDGER: OVERRIDE level=2 2.50% $25.0000 (visible to admin only)

  Visibility Engine (get_visible_agent_ids):
    Uses SQLite recursive CTE to collect all downlines of a viewer.
    Returns: [viewer_id] + all_downline_ids
    Applied as: WHERE agent_id IN (?, ?, ...) on all ledger queries.
    Admin bypasses filter entirely.

================================================================================
  9. AGENT HIERARCHY SYSTEM
================================================================================

  Design: Directed Graph (not a tree)
  - Stored as edge table (agent_hierarchy), not parent_id column on agents
  - One agent can have multiple uplines (e.g. shared MGA)
  - Transitive relationships discovered by BFS at query/write time
  - Only direct edges stored; no denormalized path cache

  Cycle Prevention:
  - On POST /api/hierarchy/link: before inserting, run get_visible_agent_ids
    on the downline agent. If the proposed upline is already a descendant
    of the downline, return 400 "This link would create a cycle."

  Demo Hierarchy (5 agents):

    Alex Morgan (admin/MGA) — agent-001
    ├── Jordan Rivera (agent) — agent-002  [5% override to Alex]
    └── Rachel Chen (agent) — agent-003    [5% override to Alex]
        ├── Marcus Webb (agent) — agent-004 [3% override to Rachel]
        └── Sofia Park (agent) — agent-005  [3% override to Rachel]

  BFS from Marcus (agent-004) when issuing a policy:
    Level 0: Marcus     → BASE (10%)
    Level 1: Rachel     → OVERRIDE (3% or rule-based)
    Level 2: Alex/MGA   → OVERRIDE (5% or 2.5% for Whole Life Advantage)

  Commission Rules (product-specific overrides):
    Product: Whole Life Advantage (prod-003)
      Level 1 upline: 4.0% (instead of 3% from edge)
      Level 2 upline: 2.5% (instead of 5% from edge)

================================================================================

  Platform: Render.com (free tier)
  URL: https://insuredeck.onrender.com

  Deployment process:
  1. git push origin main
  2. Render detects push via GitHub webhook
  3. Render runs: pip install -r requirements.txt
  4. Render runs: python app.py
  5. On startup: init_db() creates all tables (IF NOT EXISTS — safe to re-run)
  6. If clients table is empty → seed() runs automatically
  7. Flask serves React SPA from frontend/dist/ for all non-API routes

  requirements.txt:
  - flask
  - flask-cors
  - PyJWT

  Vite build output (frontend/dist/) is committed to Git so Render doesn't
  need to run npm build. The dist/ folder is excluded from .gitignore for
  this project.

  Environment variables used:
  - PORT  (set by Render automatically)
  - RENDER (set by Render, disables Flask debug mode)
  - JWT_SECRET (hardcoded for demo; should be env var in production)

  CORS allowed origins:
  - http://localhost:5173 (Vite dev)
  - http://127.0.0.1:5173 (Vite dev)
  - https://insuredeck.onrender.com (production)

  13. KEY DESIGN DECISIONS
================================================================================

  1. SQLite over PostgreSQL
     SQLite's WAL mode handles concurrent reads well for a single-server
     deployment. Zero operational overhead — no separate database process.
     Row factory + json-auto-decode makes it behave like a document store
     for JSON columns.

  2. Graph edge table (not parent_id column) for hierarchy
     An agent can have multiple uplines (e.g., shared MGA). A parent_id
     column on the agents table would only support a tree. The edge table
     supports arbitrary DAG structures with transitive BFS traversal.

  3. Immutable commission ledger (INSERT only)
     Every earning is permanent. No UPDATE path exists on commission_ledger.
     Corrections are new rows (reversal pattern). This satisfies:
     - E&O (Errors & Omissions) audit requirements
     - State DOI compliance expectations
     - Dispute resolution with verifiable paper trail

  4. Backend-enforced visibility (not UI masking)
     get_visible_agent_ids() runs a recursive CTE and returns a list.
     The WHERE agent_id IN (...) clause filters at the SQL level.
     The amount field is simply absent from the JSON response for restricted
     rows — it is never sent to the browser at all, not just hidden in CSS.

  5. Zero-disruption ledger migration
     The existing commissions table and all callers (routes, frontend) were
     left completely untouched. commission_ledger is purely additive.
     backfill_ledger() seeds historical ledger rows from existing data.

  6. Atomic commission writes
     Both the commissions INSERT and all commission_ledger INSERTs share
     the same uncommitted transaction in transition_policy(). Either all
     succeed (db.commit()) or all roll back on exception.

  7. Decimal precision for money
     Python's decimal.Decimal with ROUND_HALF_UP at 4 decimal places
     prevents floating-point rounding errors in commission calculations.
     Stored as SQLite REAL (IEEE 754 double), with 4dp rounding applied
     before storage.

  8. BFS max depth (MAX_LEVELS = 5)
     Prevents runaway traversal in deeply nested or malformed hierarchies.
     In practice, US insurance rarely exceeds 3 levels (Agent→Manager→MGA).

  9. React Flow (@xyflow/react v12) for hierarchy visualization
     Chosen for its React-native integration, built-in pan/zoom/minimap,
     custom node support, and smooth-step edge routing. The BFS layout
     algorithm assigns positions based on hierarchy level — no external
     layout library needed.

  10. Zustand over Redux for state management
      Simpler API with less boilerplate. Each domain (clients, policies,
      tasks, hierarchy, ledger) gets its own slice within a single store.
      Works well with React 18 concurrent features.

  14. FEATURE TIMELINE (What Was Built in Each Phase)

  ----------------------------
  • Full client CRUD with lead scoring
  • Pipeline stage management
  • Product catalog with eligibility rules
  • Policy lifecycle (FSM with 7 states)
  • Eligibility validation on policy creation
    (age, income, premium min/max)
  • Commission tracking on policy issuance
  • Activity logging
  • DataTable, DetailPanel, StatusBadge components
  • Sidebar navigation

  PHASE 3 — Stickiness & Intelligence
  ------------------------------------
  • Dashboard with task manager (overdue highlights)
  • Task CRUD (priority, due dates, client/policy links)
  • Needs Analysis worksheet + product recommendation engine
  • Lead source tracking (8 categories)
  • Lead scoring algorithm (profile completeness, time decay, stage velocity)
  • Analytics dashboard (MTD/YTD metrics, pipeline funnel, activity metrics)
  • Commission forecasting (pipeline probability-weighted + renewal forecast)
  • Renewal monitoring page (30/60/90 day windows)
  • 12-month commission trend (historical data)
  • US market conversion (USD, US phone formats, US state licensing)

  PHASE 4 — Agent Hierarchy + Commission Ledger (Production-Grade)
  ----------------------------------------------------------------
  • agent_hierarchy graph edge table (directed, UNIQUE, cycle-safe)
  • commission_ledger immutable table (append-only, no UPDATE ever)
  • commission_rules product-specific override schedule
  • BFS commission engine (compute_and_insert_ledger):
    - BASE entries for writing agents
    - OVERRIDE entries for all uplines (transitive)
    - Max 5 levels, visited-set cycle guard
    - Python Decimal for monetary precision
  • Visibility engine (recursive CTE):
    - Backend-enforced at SQL level
    - Amount field physically absent from payload for restricted rows
  • 7 new /api/hierarchy/* routes
  • 3 new demo agents (Rachel Chen, Marcus Webb, Sofia Park)
  • 4 hierarchy edges, 2 commission_rules
  • backfill_ledger() for existing commission records
  • HierarchyGraph.jsx (React Flow canvas):
    - Custom AgentNode (avatar, role, earnings/••••)
    - BFS layout algorithm
    - Smooth-step edges with override % labels
    - Admin slide-in link creation panel
    - Cycle detection before save
  • CommissionLedger.jsx:
    - KPI summary (BASE/OVERRIDE/total)
    - Policy flow visualizer (horizontal arrow chain)
    - Full ledger DataTable
  • Sidebar: Agent Hierarchy + Ledger nav items
